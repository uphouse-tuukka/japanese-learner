# CEO Plan: Portfolio Mini-Session

Generated by /plan-ceo-review on 2026-04-05
Branch: feature/public-challenge | Mode: EXPANSION
Repo: uphouse-tuukka/japanese-learner

## Vision

### 10x Check

Turn the public portfolio path from a one-question curiosity into a real 2-minute tutoring experience. A visitor should pick a travel scenario, get a short AI lesson, answer a few exercises, and leave with a recap that feels like a teacher noticed what they learned and missed.

### Platonic Ideal

The demo feels like a tiny, complete product loop. It starts with a real goal, teaches practical phrases in context, quizzes immediately, then closes with a smart, honest recap. No fake account theater, no random-topic drift, no toy energy.

## Product Shape

- Start screen offers a short scenario picker with 5-6 travel contexts, for example: ordering food, asking directions, checking into a hotel, shopping, transport, and greetings.
- Session stays anonymous. No guest account, no persistent learner profile, no fake public progression.
- Generation uses the real AI session planner, but targets a short public flow: 1 lesson + exactly 4 exercises.
- Public demo may use the full app exercise set if generated, including reading/listening, because the goal is to show the real product range, not a watered-down subset.
- Summary is mixed, not fake-persistent: honest session stats plus one light celebratory element such as a completion banner or accuracy badge. No XP, streaks, milestone unlocks, or level-up prompts.
- Replay policy is two completed sessions per 24 hours per visitor, enough for one retry with a different scenario without turning this into an open public product.
- Anonymous session state is server-backed, survives refresh/navigation, and resumes cleanly until completion or expiry.

## Design Direction

- This is **product-first**, not portfolio-explainer-first. The first screen should feel like entering a tiny real lesson, not reading a case study about one.
- The first viewport should be one composed experience: promise, scenario choice, and one primary CTA. No separate "how it works" strip or builder-explainer block above the fold.
- Reuse the existing app's calm Japanese visual language from `src/app.css`: warm paper surfaces, sumi text, vermillion as the main action color, matcha as secondary support, generous spacing, minimal radius, soft shadows.
- Avoid the V1 anti-patterns from the current public challenge page: split "framing column + demo card" layout, technical explainer copy competing with the task, and post-answer builder-facing notes.
- Do not drift into generic AI-product patterns: no centered SaaS hero, no 3-card feature grid, no icon bubbles, no purple gradient, no "AI-powered" marketing voice.

## First Impression Hierarchy

The visitor should understand the screen in this order:

1. **What this is:** a short Japanese travel mini-session
2. **What to do next:** choose a real travel situation
3. **Why it feels safe:** anonymous, portfolio-safe, limited public demo

```text
PORTFOLIO CHALLENGE START

[eyebrow]
Portfolio demo

[primary heading]
Take a 2-minute Japanese travel mini-session

[supporting sentence]
Pick a situation, learn a few useful phrases, then try a short guided practice.

[scenario picker]
Food | Directions | Hotel | Transport | Greetings

[primary CTA]
Start session

[quiet trust note]
Anonymous demo. No account, no permanent learner profile.
```

## Start Screen Layout

- **Desktop:** one composed start surface inside a comfortable reading width, with copy and scenario selection feeling part of the same object. The picker can sit below the promise or as a quiet two-column grid inside the same surface. Do not split the screen into a marketing column and an app column.
- **Mobile:** single-column flow with the same hierarchy, no loss of trust note, no collapsed hidden explanation needed.
- **Scenario picker interaction:** each option should feel like choosing a moment you are about to walk into, not filling out a form. Use large text buttons/tiles with concise labels, not dropdowns or tiny chips.
- **CTA behavior:** keep one obvious primary button. It stays disabled until a scenario is chosen, then becomes the only loud action on the screen.

## Screen Structure

```text
Start
  -> Lesson
    -> Exercise 1
    -> Exercise 2
    -> Exercise 3
    -> Exercise 4
  -> Summary
```

- **Start:** orient, choose scenario, begin
- **Lesson:** calm reading moment, key phrases first, no visual clutter
- **Exercises:** one exercise at a time, progress always visible, no side explainer panels
- **Summary:** recap first, stats second, celebration last

## What Already Exists

- `src/app.css` already defines the right surface, spacing, typography, and color language. Reuse it directly.
- `src/routes/learn/+page.svelte` is the best reference for pacing, loading tone, and moving from lesson to practice.
- The current `src/routes/portfolio/challenge/+page.svelte` is useful mainly as a **do not copy this layout** reference for V2.

## Not in Scope

- A marketing-style explainer section above the fold
- Builder-facing copy like "how this capsule works" inside the main learner flow
- Fake progression UI, dashboard chrome, or anything that makes the start screen feel like a public product account

## Delivery Notes

- Prefer a thin portfolio-specific orchestration layer over adapting the authenticated learn/practice routes directly.
- Reuse `generateSessionPlan()` for content generation.
- Reuse existing exercise types and rendering patterns where cleanly possible, but do not couple the public flow to authenticated gamification or user-level APIs.
- Reuse `SessionRenderer.svelte` and the existing exercise components for per-exercise interaction. Do not fork a second public-only exercise renderer unless a specific exercise type proves incompatible.
- Public listening exercises should use **browser speech synthesis**, not `/api/tts`. The portfolio flow must not depend on the public TTS endpoint.
- The public summary needs its own shape, because the existing `SessionSummary` component assumes user progression concepts that do not belong in an anonymous portfolio demo.
- Reuse the existing public-flow domain module by evolving and renaming the current portfolio orchestration file, instead of adding a second parallel V2 module beside it.
- Concretely:
  - `GET /portfolio/challenge` remains the public page shell.
  - `POST /api/portfolio/session/start` creates a server-backed anonymous session record, generates the lesson + exercises, stores them server-side, and returns a signed session token plus public payload.
  - `GET /api/portfolio/session/current` restores the active anonymous session from the signed token after refresh or return navigation.
  - `POST /api/portfolio/session/progress` persists answer state and current step after each completed exercise so refresh/resume works.
  - `POST /api/portfolio/session/complete` finalizes the anonymous session, generates the recap, applies quota accounting, and returns the public summary payload.
  - Rename and evolve the current portfolio orchestration module into the single V2 orchestration layer for start, progress, complete, quota, and fallback behavior.
  - `src/routes/portfolio/challenge/+page.svelte` owns the public state machine: intro -> loading -> lesson -> active -> summarizing -> done -> quota/error.
- Signed token means a server-issued HMAC token that references the anonymous session id and expiry.
- The signed session reference should live in a dedicated **HttpOnly active-session cookie**. The browser only keeps lightweight UI state such as selected answer or optimistic progress, never the session credential itself.
- Restore behavior means: browser reloads, the server reads the active-session cookie, then returns the current session payload. The server is the source of truth, not local storage.

## Edge Cases to Plan For

- **Generation failure:** show a clear retry state, not a broken blank screen.
- **Slow generation:** show explicit loading copy and progress framing.
- **Summary generation failure:** fall back to deterministic recap from exercise results and taught phrases.
- **Refresh or navigate away mid-session:** restore the in-flight anonymous session from local state if still valid.
- **Quota exhausted:** tell the visitor they have used today’s public demo allowance and invite them back later.
- **Expired or stale token:** fail closed, explain that the session expired, and let the visitor start over if quota allows.
- **Abandon after lesson or mid-exercise:** preserve progress until expiry so the flow feels durable, not brittle.

## Concrete Rules

- **Visitor identity key:** existing portfolio visitor cookie for soft identity, IP hash for abuse control.
- **Session TTL:** 30 minutes from start for resume and completion.
- **Quota window:** rolling 24 hours.
- **Completion cap:** 2 completed sessions per visitor cookie per 24 hours.
- **Start cap:** 4 starts per IP per 24 hours to absorb one failed generation and one replay.
- **Token artifact:** signed session token in the HttpOnly active-session cookie contains session id, visitor id, and expiry only, never full lesson/exercise payload.
- **Progress persistence:** save after each answered exercise so refresh restores the true current step.
- **Answer evaluation:** keep the existing exercise-component grading behavior for the public flow and persist the resulting answer payloads server-side. Do not invent a second per-exercise grading system for V2.
- **Generation timeout budget:** 45 seconds max, one retry only.
- **Repair rule:** if the planner returns more than 4 exercises, trim to 4. If it returns fewer than 4 valid exercises or unsupported payloads, retry once. If retry still fails, return a polite error instead of inventing a half-broken session.
- **Summary fallback:** if AI recap fails, compute deterministic recap from taught phrases, correct/incorrect answers, and scenario.
- **One active session rule:** one in-flight anonymous session per visitor cookie. Starting a new session invalidates the previous unfinished one.
- **Idempotency:** repeated `progress` writes for the same exercise overwrite the same step; repeated `complete` calls return the existing finished summary instead of double-finishing the session.
- **Storage:** session record stores scenario, lesson payload, exercises payload, answers payload, current step, status, token metadata, and expiry, with expired rows cleaned up by the existing portfolio cleanup path or a new matching cleanup step.
- **Storage shape:** extend the existing `portfolio_challenge_attempts` row into the anonymous session record instead of creating a second session table. One public session equals one attempt row.
- **Public learner adapter:** `generateSessionPlan()` receives a fixed anonymous learner context with level `beginner`, no history, no account, and scenario-specific topic preference.
- **Browser voice capability:** the client should detect whether a Japanese browser voice is available before starting the session and send that capability with the start request.
- **Listening support rule:** listening exercises are allowed in the public flow only when the active browser reports Japanese speech-synthesis support. This keeps audio in the product without requiring `/api/tts`.
- **Exercise repair order:** validate planner output, drop invalid exercises, drop listening exercises when browser voice support is unavailable for that session, keep supported results in original order, trim to the first 4 valid exercises, retry once if fewer than 4 survive.

## API Contracts

- `POST /api/portfolio/session/start`
  - request: `{ scenario: string, supportsBrowserVoice: boolean }`
  - response: `{ ok, state, sessionId, lesson, exercises, expiresAt, remainingSessionsToday }`
- `GET /api/portfolio/session/current`
  - request: no body, server reads active-session cookie
  - response: `{ ok, state, sessionId, lesson, exercises, answers, currentIndex, expiresAt }`
- `POST /api/portfolio/session/progress`
  - request: `{ sessionId, currentIndex, answer }`
  - response: `{ ok, state, answers, currentIndex }`
- `POST /api/portfolio/session/complete`
  - request: `{ sessionId }`
  - response: `{ ok, state: 'done', summary, stats, celebration }`

## Summary Shape

- `summary`: short narrative recap
- `stats`: accuracy, totalCorrect, totalExercises, scenario, phrasesLearned
- `strengths`: phrases or patterns handled well
- `misses`: phrases or patterns missed or hesitated on
- `nextSteps`: 2-3 coaching bullets for what the tutor would teach next
- `celebration`: one light public-only reward element, for example completion banner or accuracy badge

## Public State Machine

| State       | Meaning                         | Exit                       |
| ----------- | ------------------------------- | -------------------------- |
| intro       | no active session               | start                      |
| loading     | generating lesson and exercises | lesson, error, quota       |
| lesson      | lesson visible before exercises | active                     |
| active      | exercises in progress           | active, summarizing, error |
| summarizing | completion request in flight    | done, error                |
| done        | summary visible                 | restart if replay remains  |
| quota       | daily allowance used            | wait for quota reset       |
| error       | recoverable failure state       | retry or restart           |

## Reviewer Concerns

- The selected **full app exercise set** is the one deliberate stretch in this v1. It better represents the real product, but it may pressure the 2-minute demo target and require stricter prompt shaping plus repair logic than a curated subset would.
- Public AI generation plus one retry can still feel slow under real latency. If this shows up in implementation or QA, the first fallback to consider is async/polled start or a faster deterministic backup path.
