# Portfolio Demo — Reveal & Disclaimer Design Spec

**Date:** 2026-04-05  
**Scope:** `src/routes/portfolio/challenge/+page.svelte` + `src/lib/components/PortfolioBehindTheScenes.svelte`  
**Branch:** `feature/public-challenge`  
**Status:** Approved

---

## Problem / Goal

The public portfolio demo (`/portfolio/challenge`) is a standalone, no-login experience used to showcase the application to potential collaborators and employers. Two transparency gaps exist:

1. **During the lesson** — there is no acknowledgement that AI-generated content can contain occasional errors. A visitor who spots a small inconsistency may lose confidence in the demo or the product.
2. **After completing the demo** — a visitor who finishes all exercises has no way to understand what they just experienced, how the system is built, or how the demo relates to the full application. This is a missed opportunity to communicate technical depth in a portfolio context.

**Goals:**

- Add a calm, non-alarmist AI content disclaimer visible only in the `lesson` state, below the call-to-action button.
- Add a collapsible "How this works" reveal in the `done`/summary state that explains the demo and gestures at full-application depth — without feeling like a SaaS pitch.

---

## UX Decisions

| Decision                                                          | Rationale                                                                                                                                                                                           |
| ----------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Disclaimer appears only in `lesson` state                         | It is most relevant when the user is reading AI-generated lesson content. It would be noise elsewhere.                                                                                              |
| Disclaimer is placed _below_ the `I'm ready for questions` button | The primary action is not delayed or undermined. The disclaimer is supplementary, not blocking.                                                                                                     |
| "How this works" is a collapsible reveal, not always-visible copy | The summary state already carries emotional weight (score, celebration, feedback). The reveal must not compete with it. A user who wants technical context can open it; others move on undisturbed. |
| Reveal appears only in `done` state, never earlier                | Explaining the system before the user has experienced it would be premature and disorienting. Completion is the natural moment for reflection.                                                      |
| Reveal is collapsed by default                                    | Respects the calm aesthetic. Curiosity-driven — not imposed.                                                                                                                                        |
| Product-first tone, no SaaS marketing language                    | The audience is technical (portfolio visitors). Straightforward factual explanation reads as more credible than promotional copy.                                                                   |
| Medium depth only                                                 | One flow diagram plus short explanation sections. Enough to demonstrate architectural thinking; not a full technical writeup.                                                                       |

## Component Structure

### `src/routes/portfolio/challenge/+page.svelte`

**Lesson state** (`viewState === 'lesson'`):

```
<section class="card lesson-card">
  …lesson topic, explanation, cultural note, key phrases…
  <button …>I'm ready for questions</button>
  <p class="trust-note">                          ← disclaimer (new / confirmed)
    Lesson and exercise content is generated by AI
    and may occasionally contain small errors or inconsistencies.
  </p>
</section>
```

The `.trust-note` class already exists in the stylesheet and styles the element at `--text-sm` / `--text-bokashi` — no new styles required.

**Done state** (`viewState === 'done'`):

```
<section class="card summary-card">
  …celebration, accuracy, summary text, stats, strengths, misses, next steps…
  {#if remainingSessions > 0}
    <button …>Try another scenario</button>
  {:else}
    <p class="done-note">You've completed today's demos.</p>
  {/if}
  <PortfolioBehindTheScenes />              ← reveal component (new / confirmed)
</section>
```

`<PortfolioBehindTheScenes />` is placed after all session feedback content and after the CTA/done-note block, so it does not compete with primary actions.

---

### `src/lib/components/PortfolioBehindTheScenes.svelte`

Self-contained Svelte 5 component. Owns its own open/closed state via `$state`. No props required.

**Internal structure:**

```
<aside aria-label="How this demo works">
  <button aria-expanded={isOpen} aria-controls="bts-reveal">
    How this works  [chevron]
  </button>

  {#if isOpen}
    <div id="bts-reveal" role="region">
      <section>  What just happened        </section>
      <div>      Demo flow diagram          </div>
      <section>  How the demo works         </section>
      <hr />
      <section>  In the full application    </section>
    </div>
  {/if}
</aside>
```

The component is fully self-contained — it imports nothing and exports nothing. It does not interact with page state.

## Content Structure

### Disclaimer (lesson state)

> Lesson and exercise content is generated by AI and may occasionally contain small errors or inconsistencies.

- One sentence, sentence case.
- No emoji, no icon, no visual weight.
- Rendered as `<p class="trust-note">` — same class used for the existing "Anonymous demo" note in the intro state.

---

### Reveal: `PortfolioBehindTheScenes`

#### Toggle button label

`How this works`

#### Section 1 — What just happened

Short paragraph (2–3 sentences). Explains that the visitor just completed a real AI-generated lesson. Connects the scenario choice to the content output. Establishes that this is the same system as the full application.

> You completed a real AI-generated Japanese lesson. The travel scenario you chose shaped the vocabulary, phrases, and exercises — all generated fresh by the same system that powers the full application.

#### Flow diagram

An ordered horizontal list (`<ol>`) representing the demo pipeline. Five steps, each with a kanji/emoji icon and a short label. Scrollable on narrow viewports.

| Step | Icon | Label          |
| ---- | ---- | -------------- |
| 1    | 🗺️   | Scenario       |
| 2    | 筆   | AI lesson plan |
| 3    | 言   | Key phrases    |
| 4    | 練   | Exercises      |
| 5    | 完   | Summary        |

Steps are connected by a CSS-generated horizontal rule (`::after` pseudo-element on each non-last item).

#### Section 2 — How the demo works

Bulleted list (4 items). Factual, present-tense, no marketing language.

1. You chose a travel scenario from six real-world situations.
2. An AI tutor generated a contextual lesson with practical phrases and four practice exercises, tailored to that scenario.
3. Each exercise used the same components and grading logic as the full application.
4. After completing all exercises, an AI-generated summary assessed your strengths and areas to revisit.

#### Divider

`<hr>` — visual separator between "this demo" and "the full application".

#### Section 3 — In the full application

Bulleted list (4 items). Each item starts with a bolded short label. Describes capabilities absent from the demo.

1. **Adaptive learning** — the tutor remembers what you've practiced and where you struggled, shaping each new session around your actual progress.
2. **Category rotation** — exercises rotate across translation, reading, listening, fill-in-the-blank, and reordering to build balanced skills over time.
3. **Progression** — ink points, milestones, and level advancement track your growth across sessions.
4. **Conversational missions** — guided dialogues let you practice real travel conversations with an AI partner.

Bullet markers for this section use `--accent-matcha-soft` to visually differentiate it from the demo explanation above.

## Styling / Design Notes

All values reference existing design tokens from `app.css`. No new tokens needed.

### Disclaimer (`.trust-note`)

| Property     | Token                                                                                         |
| ------------ | --------------------------------------------------------------------------------------------- |
| `font-size`  | `--text-sm`                                                                                   |
| `color`      | `--text-bokashi`                                                                              |
| `margin-top` | implicit — no extra margin class needed; the existing `.lesson-card` grid gap handles spacing |

The class is already defined and shared with `.done-note` in the portfolio page stylesheet.

### `PortfolioBehindTheScenes` component

| Element                    | Styling notes                                                                                                              |
| -------------------------- | -------------------------------------------------------------------------------------------------------------------------- |
| `<aside>`                  | `margin-top: var(--space-6)` — visual breathing room from the content above                                                |
| Toggle button              | Ghost style — no border, no background; `color: --text-usuzumi`; hover: `color: --text-bokashi`, `background: --bg-washi`  |
| Chevron                    | `‹` rotated −90° (closed) / +90° (open) via CSS `transform`; animated with `--duration-normal`                             |
| Reveal container           | `background: --bg-washi`, `border: 1px solid --border-light`, `border-radius: --radius-lg`, `padding: --space-5 --space-6` |
| Reveal entrance            | `animation: revealIn` — fade + 6 px upward translate, `--duration-slow`                                                    |
| Section headings           | `text-transform: uppercase`, `letter-spacing: --tracking-wide`, `font-size: --text-xs`, `color: --text-usuzumi`            |
| Body text / list items     | `font-size: --text-sm`, `color: --text-bokashi`, `line-height: --leading-normal`                                           |
| Flow diagram background    | `--bg-kinu` — slightly deeper than the reveal container to create a nested inset feel                                      |
| Step icon nodes            | `2rem × 2rem`, `background: --bg-shoji`, `border: 1px solid --border-light`, `border-radius: --radius-md`                  |
| Connector lines            | CSS `::after`, `height: 1px`, `background: --border-mid`                                                                   |
| Full-app list markers      | `color: --accent-matcha-soft` (via `li::marker`)                                                                           |
| Full-app `<strong>` labels | `color: --text-sumi`, `font-weight: --weight-medium`                                                                       |
| Divider `<hr>`             | `border-top: 1px solid --border-light`, no vertical margin beyond grid gap                                                 |
| Mobile breakpoint          | `≤ 37.5rem` — reduce reveal padding to `--space-4`; reduce flow step padding                                               |

The flow diagram must be horizontally scrollable on narrow viewports (`overflow-x: auto`, `-webkit-overflow-scrolling: touch`, inner list `min-width: max-content`).

## State Placement / Behavior

| Behaviour              | Specification                                                                                                                                                                                                            |
| ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Disclaimer visibility  | Rendered only when `viewState === 'lesson'`. Not present in any other state.                                                                                                                                             |
| Reveal visibility      | Rendered only when `viewState === 'done'`. The component is conditionally included in the template — it is not mounted at all in other states.                                                                           |
| Reveal open/closed     | Managed by `isOpen = $state(false)` inside `PortfolioBehindTheScenes`. Starts closed. Toggled by button click.                                                                                                           |
| Reveal re-mount        | The `done` state is entered once per session. If the user clicks "Try another scenario" and completes a second session, `viewState` returns to `done` and a fresh `PortfolioBehindTheScenes` mounts (closed by default). |
| No persistence         | `isOpen` is not persisted to `localStorage` or session storage. Each page load and each session completion starts with the reveal closed.                                                                                |
| No transition on close | The reveal panel has an entrance animation (`revealIn`) but no exit animation. Exit is an immediate DOM removal via `{#if isOpen}`. This avoids layout shift artefacts when collapsing.                                  |
| Focus management       | No programmatic focus management required. The toggle button retains focus naturally after click; screen readers can navigate into the revealed region via `aria-controls` / `role="region"`.                            |

## Validation / Acceptance Criteria

### Disclaimer

- [ ] The disclaimer text appears below the "I'm ready for questions" button in the `lesson` state.
- [ ] The exact text matches: _"Lesson and exercise content is generated by AI and may occasionally contain small errors or inconsistencies."_
- [ ] The disclaimer is not visible in the `intro`, `active`, `summarizing`, `done`, `quota`, or error states.
- [ ] The disclaimer renders at `--text-sm` / `--text-bokashi` — visually subordinate to the primary button.
- [ ] The disclaimer does not shift or obscure the primary button on any viewport width.

### Reveal component

- [ ] The "How this works" toggle button is visible in the `done` state below the session feedback and CTA/done-note.
- [ ] The toggle is not visible in any other state.
- [ ] The reveal is closed by default on page load and on each new session completion.
- [ ] Clicking the toggle opens the reveal panel with an entrance animation.
- [ ] Clicking the toggle again closes the panel immediately (no exit animation).
- [ ] `aria-expanded` on the toggle reflects current open/closed state.
- [ ] The reveal panel has `role="region"` and an `aria-label`.
- [ ] The flow diagram renders all five steps in order: Scenario → AI lesson plan → Key phrases → Exercises → Summary.
- [ ] The flow diagram is horizontally scrollable on viewports ≤ 375 px without clipping step labels.
- [ ] Section order inside the reveal is: _What just happened_ → flow diagram → _How the demo works_ → divider → _In the full application_.
- [ ] Full-application bullet markers are visually distinct from demo-section bullets (`--accent-matcha-soft`).
- [ ] No new design tokens are introduced; all styles use existing `app.css` variables.
- [ ] Component renders correctly after "Try another scenario" and completing a second session.
